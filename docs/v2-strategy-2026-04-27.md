**Status:** Draft for review. No code changes proposed by this document — only the *specs* that subsequent code changes would implement.

**Inputs:** the literary audit at `docs/audit-2026-04-27.md` (2026-04-27).

**Goal of this document:** provide three concrete artifacts that, once approved, drive the actual implementation work:

1. A v2 word-ladder for stories 1–10 with literary north-stars.
2. Exact specs for 4 new/tightened deterministic lint rules.
3. Spec for a literary-review gate (the "engagement reviewer" referenced in `semantic_lint.py`).

## 1. The v2 word-ladder for stories 1–10

### 1.1 Design principles

The audit shows defects cluster where authors had no concrete language for the scene they wanted. The fix is to seat **size**, **color**, **spatial position**, and at least one **bodily action verb** before story 6, so the corpus's first cat / first letter / first walk can be described concretely instead of via metaphor.

Three rules govern the ladder:

- **R1 — Concrete before abstract.** A noun ships before any adjective, metaphor, or relative clause that depends on it. Example: 花 ships before 花の色 is allowed.
- **R2 — Every story introduces ≤ 6 new words and ≤ 1 new grammar point.** Matches current pipeline norms; keeps the cognitive load constant.
- **R3 — Every story has at least one *concrete action verb* available.** Not just observation verbs. This is what was missing in stories 6/11/12/13 — the narrator could only *look at* and *think about*; never *put down*, *open*, *sit*.

### 1.2 The proposed ladder (vs. current)

| Story | Current new_words | Proposed new_words | Δ rationale |
|---|---|---|---|
| 1 | 今朝, 雨, 私, 窓, 外, 見ます, 木, 濡れる, お茶, 飲みます, 静か, 暖かい, いい, 気分 | unchanged (already 14 — the bootstrap) | keep — story 1 is solid |
| 2 | 朝, 公園, 歩きます, 夕方, 朝ご飯, 卵, 食べます, 友達, 散歩 | 朝, 公園, 歩きます, 夕方, **小さい**, 友達, 散歩 | drop 朝ご飯/卵/食べます (move to 3), introduce **小さい** (W00055) so the scene noun-pile defect is preventable from story 3 onward |

### 1.3 What this ladder *prevents* by construction

- **No tautological-equivalence in stories 1–6** — 色 isn't introduced until story 7, and by then there are concrete colors, so `Xの色は Yの色です` is dominated by `Xは赤いです` / `Xは白いです`.
- **No noun-pile-closer crutch** — every story 5+ has a concrete action verb available, so closers can be `〜ます` sentences.
- **No misapplied-quiet-on-rain** — 静か stays available but story 8's 明るい gives night scenes a non-quiet adjective to lean on.
- **No bare-known-fact crutch from story 10** — 言います arrives, so dialogue replaces 思います as the default psychological move.

### 1.4 Required new vocabulary (3 words)

The ladder requires 3 new mints — they don't currently exist in `data/vocab_state.json` and they're load-bearing:

- **下 (した, "below / underneath", noun)** — for story 3
- **置きます (おきます, "to place / put down", verb)** — for story 6
- **持ちます (もちます, "to hold / carry", verb)** — for story 9

Each is a Genki Lesson 1–3 word. Adding them to vocab is a one-line append per word + occurrences will accrue naturally.

### 1.5 Literary north-stars for stories 1–10

A north-star is one hand-authored sentence per story that the rest of the story has to live up to. It's *not* a constraint — it's a *template for voice*. Authors (human or LLM) read the north-star and use it as the standard for the story's tone and concreteness.

| Story | North-star (proposed) | What it teaches downstream authors |
|---|---|---|
| 1 | 「窓の外で、雨が木を濡らします。」 | Concrete subject + action; no pathetic fallacy needed |
| 2 | 「友達と公園を歩きます。小さい花があります。」 | Walking *with* someone; named smallness instead of vague "quiet" |
| 3 | 「机の下に、卵があります。」 | Spatial anchor (under the desk); a concrete found object |
| 4 | 「ドアの上に、赤い花があります。」 | Named color above a doorway — eliminates "the door's color is the flower's color" |
| 5 | 「犬は椅子に座ります。」 | The first sit-down; physical action by a non-narrator |
| 6 | 「猫は窓のそばにいます。私はペンを置いて、手紙を書きます。」 | Two grounded actions in one beat; the cat *exists in space*, not as a metaphor |
| 7 | 「月は白いです。星も白いです。」 | Direct color description — replaces "moon's color = star's color" tautology |
| 8 | 「夜の窓は明るいです。私は本を読みます。」 | A night that is bright, not just quiet |
| 9 | 「猫は私のそばで寝ています。私は本を持ちます。」 | Two beings, one room, both doing something concrete |
| 10 | 「友達は『元気ですか』と言います。」 | Dialogue arrives. From here on, 思います is for genuine inference |

### 1.6 What about stories 11–56?

This document deliberately stops at story 10. The audit shows that fixing the *opening* fixes most of the cascade — once concrete language is available in the bootstrap, the patterns the corpus learns to imitate are the right ones. Stories 11+ get re-evaluated *after* the ladder is in place, and the projected damage list is in the audit (§1: 21 REWRITE-STORY entries; the proposal in §4 below trims this further).

## 2. Lint-rule specs (4 rules)

Each rule below is specified to the level needed to write the implementation. All rules live in `pipeline/semantic_lint.py`. Each follows the existing `Issue` dataclass shape. Rules are *conservative by default* — they fire only on patterns that have direct evidence from the audit; false positives are worse than false negatives.

### 2.1 Rule 11.6 — `closer_noun_pile` (severity: error)

**Pattern:** the *last* sentence of a story (highest sentence idx) consists of:
- ≥ 2 content nouns connected by `や` / `と` / `、`
- optionally an adjective modifying the last noun (`Adj N`)
- terminated by `です` / `だ`
- and **no verb anywhere in the sentence except the final copula**

**JSON token-level test:**
```
def is_closer_noun_pile(sentence_tokens):
    has_verb = any(t.get('role') == 'content' and is_verb_pos(t) for t in tokens)
    if has_verb: return False
    last_non_punct = [t for t in tokens if t.get('role') != 'punct'][-2:]
    if last_non_punct[-1]['t'] not in ('です','だ'): return False
    content_nouns = [t for t in tokens if t.get('role')=='content' and is_noun_pos(t)]
    if len(content_nouns) < 2: return False
    list_particles = [t for t in tokens if t.get('t') in ('や','と','、')]
    if len(list_particles) < 1: return False
    return True
```

**Apply only to** the highest-idx sentence in the story. Mid-story noun-pile sentences are sometimes intentional (rhetoric / scene-setting) and shouldn't fire.

**Allowed escape hatches** (to avoid false positives):
- If the sentence contains the grammar id `G009_mo_also` (も) — likely deliberate parallelism, not a pile.
- If the corpus author has marked the story with `"closer_intentional": true` in the bilingual spec — author override, but logged.

**Negative examples (must NOT fire):**
- `そして、猫は窓から私を見ます。` — has a verb, fine.
- `朝、雨と風です。` — *opener*, not closer; rule restricted to highest idx.
- `家にも本があります。` — has `あります` (verb), fine.

### 2.2 Rule 11.7 — `tautological_possessive_equivalence` (severity: error)

**Pattern:** sentence matches the surface regex (after token reconstruction):
```
<word_id A> の <word_id Y> は <word_id B> の <word_id Y> です
```
where Y is the **same word_id on both sides** of は, and A ≠ B.

**Token-level test:**
```
# Walk tokens; find each subsequence:
#   content_noun_A, の(possessive), content_noun_Y, は(topic),
#   content_noun_B, の(possessive), content_noun_Y, です/だ
# where word_id of Y is identical at both occurrences.
```

**Allowed escape hatches:**
- If Y is `名前` (name) — comparing two people's names is meaningful (`私の名前は父の名前です` = I'm named after my father), so Y ∈ {名前, 仕事, 家} should warn-not-error or be exempt.
- If the second clause adds an adjective (`猫の色は、明るい雨の色です`) — the adjective makes it an actual evaluation. Skip.

**Negative examples (must NOT fire):**
- `猫の名前は父の名前です。` — meaningful identity claim.
- `猫の色は赤いです。` — not a possessive equivalence, just an attribute.
- `猫の色は雨の明るい色です。` — adjective adds content.

### 2.3 Rule 11.1 (loosened) — `inanimate_quiet_extended`

**Current rule** fires only on `<inanimate> は 静か です/だ` adjacent. **Loosen to also fire on:**

- `<inanimate> は 静か<copula-distance> です/だ` (allow particles between, e.g. も, さえ).
- `<inanimate> + 静かに + verb` — adverbial form. Fire when the inanimate noun is the *subject* of the adverb (no animate noun intervenes).
- `静かな <inanimate> です/だ` — when the inanimate noun is at sentence-final position and is one of the audit's flagged nouns (rain, moon, stone, letter, abstract noun).

**Add to `INANIMATE_QUIET_NOUN_IDS`:**
- W00027 (風 / wind) — wind cannot be quiet in JP
- 心の中の気持ち pattern (abstract emotion-noun phrase) — generic abstract-noun heuristic

**Allowed escape hatches:** keep the existing carve-out for window/room/place. Add 道 (road) only when modified by 静かな (a quiet road is fine; "the road is quiet" said as a bare assertion is the defect).

**Negative examples (must NOT fire):**
- `部屋は静かです。` — room is allowed.
- `静かな朝です。` — adjective on the time-of-day setting; fine.
- `猫は静かに寝ています。` — animate subject; fine.

### 2.4 Rule 11.3 (loosened) — `bare_known_fact_extended`

**Current rule** requires the embedded clause to be exactly one bare noun. **Loosen to also fire when:**

- The と-clause is a *full sentence* whose subject + predicate match a fact already asserted (with `です` or as an `あります/います` existence claim) earlier in the same story.
- Specifically catches: `〜と思います` where the embedded sentence is `<known-noun-from-prior-sentence> + <copula or です-adjective>`.

**Algorithm sketch:**
```
1. Build a set of "asserted facts" per story: for each sentence with a plain
   declarative ending (です/だ/ます), record (subject_wid, predicate_wid_or_adj).
2. For each 〜と思います sentence, parse the embedded clause; if its
   (subject, predicate) is a member of the asserted-facts set OR if the
   subject is a sensorily-immediate noun (朝, 夜, 雨, 風, 雪) the rule fires.
```

**Escape hatches:**
- If the と-clause introduces *new evaluation* (an adjective not previously used on the noun, e.g. `雨は冷たいと思います` after `雨です`) — fine.
- If the と-clause expresses a **future or counterfactual** (`明日は雨だと思います`) — fine.

**Negative examples (must NOT fire):**
- `明日は雨だと思います。` — future inference.
- `猫は元気だと思います。` — emotional state of another being is genuinely inferred.

## 3. The literary-review gate (defect 2.5: `missing-arc`)

The four lint rules above cannot catch missing-arc; it's a story-level structural problem, not a sentence pattern.

### 3.1 Spec

A new module `pipeline/literary_review.py` exposes:

```
def review_story(story_dict, *, model="gpt-mini") -> ReviewResult
```

Where `ReviewResult` has:
- `verdict: "ship" | "rewrite-sentence" | "rewrite-story"`
- `defects: list[Issue]` — same shape as semantic_lint
- `arc_score: int 0..3` — 0 = no arc, 3 = beginning/middle/end clearly present
- `north_star_alignment: int 0..3` — does the story's tone match the era's north-star sentence?
- `notes: str` — short prose explanation

The reviewer prompt includes:
1. The full story (JP + EN gloss).
2. The era's north-star sentence (from `data/north_stars.json` — new file, derived from §1.5).
3. The five-defect rubric from the audit (§2 of `audit-2026-04-27.md`).
4. The *negative* examples from each lint rule's escape hatches (so the reviewer doesn't over-flag).

### 3.2 When it runs

- In `regenerate_all_stories.py`: after a story's tokens are built but before it's written to `stories/`. Failure = block.
- In CI: warning only (so existing library doesn't suddenly break the build).
- Cached by content hash to avoid re-reviewing unchanged stories.

### 3.3 Why an LLM gate is acceptable here

The audit was performed by LLM editors and the calls converged. This is empirical evidence that an LLM can detect missing-arc and friends with reasonable consistency given the right rubric. Determinism comes from caching (same input → same output via hash) and from the rules being conservative-by-default.

## 4. Migration & cascade analysis

If this strategy is approved, the implementation order is:

### 4.1 Phase 3a — Lints first (zero cascade)
1. Implement rules 2.1, 2.2 in `semantic_lint.py`. Add tests using sentences from the audit as positive cases and SHIP-list sentences as negative cases.
2. Loosen rules 11.1, 11.3 with the same test treatment.
3. Run `python3 pipeline/validate.py` across all 56 stories. **Expected output:** the audit's 5 categories now fire as errors (24 stories or so light up). This is the "real" rewrite list — narrower than the audit's full REWRITE-STORY count because rule 2.5 (missing-arc) is excluded for now.
4. Disable the new rules in CI (warning-only) so the existing library still passes until rewrites land.

### 4.2 Phase 3b — Word-ladder reorder (bounded cascade)
1. Mint 3 new vocabulary entries: 下, 置きます, 持ちます.
2. Re-author bilingual specs for stories 2–10 to match §1.2's new word distribution (using §1.5 north-stars as drafting templates).
3. Run `regenerate_all_stories.py --apply` from story 2.
4. **Expected cascade:** stories 11+ that introduced any of {小さい, 赤い, 白い, 明るい, そば, 座ります, 言います} as `is_new` need their `new_words` list recomputed (free, automatic via `regenerate_all_stories`). Substantive content in stories 11+ does not need re-authoring unless they're already on the rewrite list.
5. Audio for stories 2–10 needs rebuild — bounded cost.

### 4.3 Phase 3c — Targeted rewrites (the actual editorial work)
1. Take the 24 REWRITE-STORY entries from the audit, intersect with the new lints' actual fire list — likely 12–18 stories.
2. Cull aggressively: which of these earn their slot? Decide whether to rewrite or to retire (renumber surviving stories).
3. Hand-author or LLM-draft (gated by the literary reviewer from §3) the surviving rewrites.
4. Apply 17 single-sentence edits for the REWRITE-SENTENCE entries.

### 4.4 Phase 3d — Engagement reviewer
1. Implement `pipeline/literary_review.py` per §3.
2. Backfill review records for the entire library.
3. Wire into the pre-commit hook and CI.

### 4.5 Estimated total effort

| Phase | Effort | Cascade size |
|---|---|---|
| 3a — Lints | ~0.5 day | 0 stories changed |
| 3b — Reorder | ~1 day | ~10 stories regenerated, no content rewrite |
| 3c — Rewrites | ~2–6 days (depends on cull aggressiveness) | bounded by §4.3 cull |
| 3d — Reviewer | ~1 day | 0 stories changed; gates future work |
| **Total** | **~5–10 working days** | **deterministic and bounded** |

Compare to "rebuild from scratch": **months** of work, *and* it starts you back at the same problem if §3d isn't built first.

## 5. What this strategy explicitly does NOT do

To keep this proposal honest:

- **It does not promise the corpus will be "literary."** It promises the corpus will *not produce specific defect categories* and that there will be a gate against ones we discover later. Literary quality remains a human judgment.
- **It does not guarantee zero false positives.** Every conservative rule misses some defects; every aggressive rule misclassifies some valid prose. The escape hatches in §2 are calibrated against the audit but will need tuning.
- **It does not fix all 21 REWRITE-STORY entries automatically.** It makes the *next* story author less likely to repeat the failures, and it bounds the existing failures so they can be triaged rather than infinitely scanned.
- **It does not solve grammar progression.** That's a separate axis; this document is about literary quality. The two interact (e.g. introducing 言います earlier changes the grammar progression downstream) but the v2 ladder respects all current grammar dependencies.

## 6. Open questions for the human

Before any code is written, the following decisions should be made by you:

1. **Cull policy.** When the lints fire on a story, default to *rewrite* or default to *retire*? Both are valid; they imply different end-state corpus sizes (~56 vs ~38).
2. **Reviewer model.** §3 assumes an external LLM call. Is that acceptable for the pipeline (latency, cost, determinism), or should the reviewer be limited to deterministic rules + a manual gate?
3. **North-star authorship.** Are the §1.5 north-stars acceptable as drafted, or do you want to author them yourself? They drive everything downstream.
4. **Backwards compatibility.** The word-ladder reorder will change `new_words` on stories 2–10 in a visible way. Do existing learners who completed stories 1–8 of the old order matter, or is a clean break acceptable?
5. **Story 1.** I left story 1 unchanged. Do you want to revisit (`雨は静かです` is its only flagged sentence and many natives accept it as standard pathetic-fallacy)?

# Addendum — Authoring Tooling (added 2026-04-28 00:00)

## A1. What already exists

The repo has a fairly mature tooling layer that this audit initially overlooked. An honest inventory:

### `pipeline/lookup.py` — search & introspect
- `lookup.py <kanji>` — find a word in vocab.
- `lookup.py --by-kana のむ` — find by kana.
- `lookup.py --grammar te-form` — find a grammar point.
- `lookup.py --next` — what story is next; recommended size.
- `lookup.py --reuse-preflight <spec>` — what should I reinforce in this story?
- `lookup.py --catalog N5` / `--untaught N5` — JLPT-level catalog views.
- `lookup.py --grammar-usage` — grammar occurrence counts library-wide.
- `lookup.py --low-occurrences` — words used ≤ N times.
- `lookup.py --by-surface` — find every story containing a surface form.

### `pipeline/precheck.py` — fast preflight
Lightweight version of the validator. Surfaces the most common mistakes (missing readings, unknown surfaces, ratio violations) before the full `validate.py` is invoked.

### `pipeline/tools/` — five surgical CLIs
- `vocab.py would-mint "毎日、本を読みます。"` — **the killer feature.** Tokenize a candidate sentence and tell the author whether it would mint a new word *before* committing it.
- `vocab.py search / info / orphans / first-occurrence / range` — full inventory access.
- `cadence.py vocab-reinforce / grammar-reinforce / validate / all` — run the pedagogical-cadence checks live, no pytest dance.
- `spec.py show / append / replace / delete / move / find / title` — surgical edits to the bilingual specs without find-and-replace English drift.
- `weave.py suggest / preview / apply` — assisted reinforcement-weaving when cadence violations appear.
- `regen.py` — incremental regeneration with diff/dry-run.
- `story.py` — per-story introspection (band check, content stats).

**Honest read:** the technical-correctness side of authoring is well-served. What's underused isn't missing tools; it's **enforcement** of the tools we have, and **upstream** tools that help with literacy and storytelling, not validation.

## A2. v2 enforcement — make existing tools mandatory in the workflow

The current authoring workflow (per `docs/authoring.md`) treats these tools as *available*. v2 should treat them as *required*. Concrete changes:

### A2.1 Pre-commit hook expansion
Currently `.githooks/pre-commit` runs … (need to verify; assume basic validation). Add:
1. **`vocab.py would-mint`** must pass on every changed bilingual spec — block commit if it would silently mint words.
2. **`cadence.py all`** must pass — block commit if pedagogical cadence drifts.
3. **`lookup.py --reuse-preflight`** must run on any newly added story spec — output is appended to the commit message as an audit trail.

### A2.2 Authoring template (new)
A new file `pipeline/inputs/_template.bilingual.json` with inline comments listing the tools the author *must* run, in order, before the spec is considered ready. The `spec.py` tool gets a `new` subcommand that copies this template and pre-fills the next story id.

```bash
spec.py new                       # creates inputs/story_N+1.bilingual.json from template
spec.py preflight 7               # runs would-mint + reuse-preflight + precheck on story 7
```

### A2.3 Authoring guide rewrite
Replace `docs/authoring.md` step 1 ("Write the bilingual spec") with a *script*:
1. `spec.py new`
2. For each candidate sentence: `vocab.py would-mint "<sentence>"` → reject if mint count > 0 unless author intentionally minting.
3. `lookup.py --reuse-preflight inputs/story_N.bilingual.json` → the tool tells you which existing words/grammar are due for reinforcement; **prefer** sentences that hit those.
4. `spec.py preflight N` → must pass before regen.
5. Then existing steps 2–5 (regen/validate/audio/ship).

This eliminates the "I forgot to check what's overdue for reinforcement" failure mode that produces stories built entirely from convenient-but-overused words.

## A3. New tools to propose for v2

These targeted additions buy disproportionate brain-power-back for the *literary* side of authoring. Each is small (≤200 lines) and composes with what exists.

### A3.1 `tools/palette.py` — concrete palette for a story
**Purpose:** Given a story id (or "next"), produce a **literary palette** — the concrete nouns/verbs/adjectives the author can use, *grouped by sensory category* and *flagged by reinforcement debt*.

```bash
palette.py next
# Output:
#   STORY 11 — palette
#   ===================
#   Concrete nouns by category:
#     places: 公園 ★, 家 ★, 道, 駅, 部屋, ベンチ
#     objects: 本 ★★, 椅子 ★, 皿, ペン, 手紙, ドア, 窓
#     beings: 私, 友達 ★★, 猫 ★, 犬, 二人
#     time/scene: 朝, 夕方, 夜, 雨, 風, 雪
#   Adjectives:
#     size: 小さい ★, 大きい
#     color: 赤い, 白い
#     temperature: 暖かい, 寒い
#     light: 明るい, 静か (use sparingly)
#   Action verbs:
#     locomotion: 歩きます ★, 帰ります, 来ます, 立ちます
#     handling:   持ちます, 置きます, 開けます
#     posture:    座ります, 寝ます
#     speech:     言います, 話します
#   Grammar points active: G001..G017
#   ★ = overdue for reinforcement (R1 violation imminent)
#   ★★ = critically overdue (R1 violation if not used in next 2 stories)
```

**Why this matters:** the audit shows authors default to over-used motifs (rain + cat + window) because they're top-of-mind. A grouped palette with *reinforcement-debt stars* nudges authors toward variety **and** pedagogical health simultaneously. This is the single highest-leverage tool we don't have.

### A3.2 `tools/north_star.py` — show the era's literary template
**Purpose:** Print the north-star sentence(s) for the era the next story belongs to. Pulls from a new file `data/north_stars.json` (populated from §1.5 of the main strategy doc).

```bash
north_star.py next
# Output:
#   STORY 11 belongs to era 11–15. North-stars:
#     - 「友達は『元気ですか』と言います。」  (story 10, dialogue arrives)
#   Voice notes:
#     - Concrete actions; direct dialogue when characters speak.
#     - One named color per story preferred over abstract 色.
#     - 静か reserved for places, not abstract feelings.
```

The reviewer agent (§3 of main doc) reads the same file — author and reviewer share a literary contract.

### A3.3 `tools/draft.py` — LLM scaffold with hard guardrails
**Purpose:** Generate a *draft* sentence given a constraint set: target words, target grammar, max content tokens, scene description. Wraps an LLM call with the v2 lints in the loop — the LLM cannot return a sentence that fails any lint rule.

```bash
draft.py --story 11 --use-words "椅子,座ります,赤い" \
         --grammar G017_de_means \
         --scene "narrator sits on a red chair by the window in the morning"
# Output (after internal retry on lint failure):
#   候補: 朝、私は赤い椅子に座ります。窓の外には雨があります。
#   gloss: In the morning, I sit on the red chair. Outside the window, there is rain.
#   ✓ would-mint: 0
#   ✓ no closer-noun-pile, no tautology, no misapplied-quiet
```

This is the "release brain power for literacy" tool. The author specifies *intent and constraint*; the tool returns a draft that's *guaranteed* mechanically valid. The author keeps the literary judgment; the LLM does the technical bookkeeping.

### A3.4 `tools/echo.py` — scan for already-used phrases
**Purpose:** Before committing a sentence, check whether its surface form (or near-paraphrase) has been used in any prior story. Prevents the rain-cat-window cycle from extending.

```bash
echo.py "猫は窓から私を見ます。"
# Output:
#   Exact match: story_6 s5, story_11 s7
#   Near-paraphrase (≥0.7 token overlap): story_3 s2, story_9 s4, story_25 s2
#   ⚠ This sentence shape ("X は Y から 私 を 見ます") has been used 7 times.
#   Suggest variation: change subject, particle, verb, or perspective.
```

Built on token-set Jaccard; cheap; catches the cycle the existing `motif_rotation_lint` misses (because that one is per-story-vocab, not per-sentence-shape).

### A3.5 `tools/arc.py` — sentence-arc visualizer
**Purpose:** Given a story spec, plot the sentence arc as a single-line ASCII diagram so the author can see whether the story has movement.

```bash
arc.py 16
# Output:
#   STORY 16 — 庭のパン
#   s0  setting   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (kitchen, morning)
#   s1  action    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (bake bread)
#   s2  action    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (make tea)
#   s3  arrival   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (bird arrives) ★ inflection
#   s4  action    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (share bread)
#   s5  reflection━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (warmth/quiet)
#   s6  closer    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (continued companionship) ★ payoff
#   ✓ Arc detected: setting → action → inflection → reflection
#
# vs.
#   STORY 6 — 猫
#   s0  setting   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s1  observation━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s2  observation━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s3  observation━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s4  observation━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s5  observation━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   s6  tautology  ━━━━━━━━━━━━━━━━━━━━━━━━━━━ ✗
#   s7  noun-pile  ━━━━━━━━━━━━━━━━━━━━━━━━━━━ ✗
#   ⚠ No inflection point. Six observations + closure-as-decoration.
```

Classification heuristic: tag each sentence by syntactic shape (existence claim / action / dialogue / reflection / closer-pile / tautology) using the lint-rule machinery already in §2. Authors can *see* whether their story has narrative movement before they ship.

### A3.6 `tools/coverage.py` — what's the corpus missing?
**Purpose:** Report which scenes / scenarios / verb classes the corpus *hasn't* depicted. Helps avoid yet another rainy-morning story.

```bash
coverage.py scenes
# Output:
#   Scene types depicted (count):
#     morning at home: 14 stories
#     evening park:    8 stories
#     letter writing:  6 stories
#     train station:   2 stories
#     bookstore:       7 stories
#     market/shop:     1 story  ★ underused
#     classroom:       0 stories  ★★ never depicted
#     workplace:       0 stories  ★★ never depicted
#     kitchen (active cooking, not just tea): 2 stories  ★ underused
```

Built on the noun-co-occurrence patterns the audit already classified.

## A4. Tools-as-policy: the principle

The trap with technical tooling is the same one the audit caught with lints: **tools you don't enforce are tools you don't have.** v2 should commit to:

1. **Every authoring action that can be checked, must be checked.** The pre-commit hook is the enforcement point.
2. **The author should specify *intent*; tooling fills in *technicalities*.** `palette.py`, `north_star.py`, `draft.py`, `echo.py`, and `arc.py` together let the author think in terms of "what story do I want to tell" while the tools handle "is this word in budget / does this break cadence / has this shape been used before."
3. **The author always has override authority** — the lints have escape hatches (§2.1's `closer_intentional`); the tools have `--ignore` flags. But every override leaves a trail in the spec, so future audits can find them.
4. **Tools and reviewers share rules.** The literary-review gate (§3) reads the same defect rubric the lints fire on, plus the north-stars the palette tool recommends. No author–reviewer rule drift.

# Addendum 2 — Author = LLM Agent (added 2026-04-28 00:05)

## B0. The audience reframe

Earlier sections of this document treated the author as a human. The actual author is an **LLM agent** (this assistant or a successor). Every section that assumed a human authoring loop needs adjustment. Specifically:

- **Pre-commit hooks for human lapses** become **agent-loop guardrails** that fail loudly and *recoverably*.
- **"Author ergonomic" tools** (ASCII art, diff visualizations, color terminals) lose value; **machine-readable JSON output** gains value.
- **`docs/authoring.md` as a how-to for humans** becomes **`docs/authoring.md` as a procedure spec for the agent** — closer to a runbook than a guide.
- **Override discipline** changes character: a human author overrides because they have judgment the rule lacks; an agent should *almost never* override, because if a rule is wrong the rule should be fixed, not bypassed.
- **Literary review** stops being "double-check the human"; it becomes "double-check the agent that wrote the story" — a *different agent* with a *different rubric* must do the review, or the system grades its own homework.

## B1. What changes about §3 (the literary-review gate)

Earlier I specified `pipeline/literary_review.py` as an LLM gate to catch missing-arc. With the author = LLM, this gate is not "nice to have"; it is **the only thing standing between the corpus and the same defect cycle the audit found.**

Adjustments:

1. **Reviewer LLM must be a different model family or a different prompt persona than the author**, with no shared scratchpad. Otherwise the system grades its own homework.
2. **The reviewer must be allowed to fail the author with a structured response** (`{"verdict": "rewrite", "defects": [...], "specific_sentence_hints": ["replace s5 with a dialogue beat", ...]}`) so the author loop can act on it without re-prompting from scratch.
3. **Cap the author↔reviewer ping-pong at 3 rounds.** If round 3 still fails, *escalate to the human* (you) with the full transcript. This is the human-in-the-loop circuit-breaker for when the agent gets stuck on a defect category neither the author-prompt nor the lints can name.
4. **Cache reviewer verdicts by the SHA of the story tokens.** Re-running the review on an unchanged story must be free.

## B2. What changes about §A2 (existing-tool enforcement)

The pre-commit hook is now misnamed. The agent doesn't `git commit`; the agent runs the pipeline. Replace the section with:

### B2.1 The agent-loop gauntlet (replaces "pre-commit hook")

A new orchestrator `pipeline/author_loop.py` that, given a story id, runs:

```
1. spec.py preflight N        → would-mint, reuse-preflight, precheck
   ↳ on fail: surface the diagnostic to the agent, do NOT proceed
2. text_to_story → regen      → token-level build
   ↳ on fail: ditto
3. validate.py + semantic_lint → all v2 lint rules including the four new
   ↳ on fail: surface specific lint messages, agent retries (max 3)
4. literary_review.py         → §3's reviewer
   ↳ on fail: surface defects + sentence hints, agent retries (max 3)
5. cadence.py all             → reinforcement debt across library
   ↳ on warn: surface debt list, agent must address in next 1–2 stories
6. audio rebuild              → only if 1–5 pass
7. write to stories/, append review record to data/reviews.json
```

This is one CLI call. It is the **only sanctioned way** for the agent to add a story. Manual edits to `stories/*.json` are forbidden in v2 (they always were de facto, but now it's policy).

### B2.2 Override discipline — narrow

The agent gets exactly one structured override: `"override": {"rule_id": "...", "rationale": "<one sentence>"}` in the bilingual spec. Each override is logged in `data/overrides.json` with timestamp, rule, rationale, and story id. **An override does not silence the rule** — it logs the violation, lets the story ship, and surfaces in `cadence.py overrides` so the human can review them periodically.

If the agent uses the same override on the same rule three times across the corpus, the rule itself is suspect; the agent **must** stop using overrides on that rule and instead surface a "this rule should be revisited" note to the human.

### B2.3 Authoring template — JSON, not prose

`pipeline/inputs/_template.bilingual.json` for an agent should be a JSON schema with required fields, not a prose template:

```json
{
  "story_id": null,
  "title": {"jp": null, "en": null},
  "north_star_consulted": null,
  "palette_consulted_at": null,
  "intent": "1–2 sentences: what story is this, why does it earn its slot",
  "scene_class": "one of: home_morning, station, walk, indoor_meal, ...",
  "anchor_object": "the concrete thing the story rotates around",
  "characters": ["narrator", "..."],
  "sentences": [{"jp": "", "en": "", "role": "setting|action|dialogue|inflection|reflection|closer"}]
}
```

The **`role`** field is new and required. It forces the agent to label each sentence's narrative function before the lints get to evaluate the result. `arc.py` (now repurposed; see B3.5) consumes this directly.

The **`intent`**, **`scene_class`**, and **`anchor_object`** fields are required because the audit showed defective stories almost universally lacked one of these *upstream*. Forcing them into the spec means the agent cannot start writing sentences before deciding what the story is *for*.

## B3. What changes about §A3 (new tools)

### B3.1 `palette.py` — STILL the highest-leverage tool, but JSON-output by default
Drop the human-pretty grouped output as the default. Add `--format human` for the rare case a human is reading. The agent reads the JSON. Same logic, different presentation.

### B3.2 `north_star.py` — now critical, not nice-to-have
The agent has no muscle memory; it cannot internalize an era's voice. The north-star file is the only thing keeping the agent from drifting toward the rain-cat-window default on every prompt. Wire it into the **author prompt**, not just an authoring CLI.

### B3.3 `draft.py` — REMOVE
This was specced as "LLM with lints in the loop." When the author *is* an LLM, this collapses into "the agent's normal workflow with the §B2.1 gauntlet." There is no separate `draft.py`; the author loop *is* the draft loop. Saves ~1 day of work.

### B3.4 `echo.py` — STILL critical
The agent has no episodic memory across sessions. Without this tool, it will re-author "the cat looks at me from the window" forever. Make this **part of the author prompt context**: every story-authoring prompt includes the top-20 most-reused sentence shapes and a "do not produce these" instruction.

### B3.5 `arc.py` — REPURPOSE, don't remove
The ASCII visualization is irrelevant for an agent. But the underlying classifier — *what role does each sentence play in the story* — is exactly what the new `role` field in B2.3 needs. Reframe `arc.py` as:

- `arc.py classify <story.json>` → outputs the inferred `role` for each sentence (JSON), so the validator can compare the agent's *declared* role against the *detected* role and flag mismatches.
- `arc.py validate <story.json>` → checks that the role sequence forms a valid arc shape (must contain at least one of {action, dialogue, inflection}; must not be 7 × observation; closer must not be tautology or noun-pile).

Drop the ASCII diagram code path entirely.

### B3.6 `coverage.py` — now load-bearing
The agent's default is to repeat itself. `coverage.py` becomes part of the author prompt context every time, surfacing "scene types you haven't touched in N stories" so the agent rotates on its own.

### B3.7 NEW — `tools/agent_brief.py`

The single most important new tool given the audience reframe. Given a story id, produces the **complete authoring context** the agent needs in one JSON payload:

```bash
agent_brief.py 11
# JSON output:
{
  "story_id": 11,
  "size_band": {"sentences": [5,9], "content_tokens": [9,38], "target": 21},
  "available_words": [...palette grouped by category, with reinforcement-debt stars...],
  "available_grammar": [...],
  "north_stars": [...for this era...],
  "scene_coverage": {"underused": [...], "overused": [...]},
  "echo_warnings": [...top-20 reused sentence shapes to avoid...],
  "reinforcement_debt": {"critical": [...], "due": [...]},
  "lint_rules_active": [...with one-line summaries...],
  "anti_patterns_to_avoid": [...the 5 audit defects + escape-hatch examples...],
  "previous_3_stories_summary": "1-line each, so the agent doesn't continue a motif"
}
```

This single JSON blob is what gets injected into the author prompt as system context. It composes the output of `palette.py`, `north_star.py`, `echo.py`, `coverage.py`, `cadence.py`, and a few others into the format the agent actually consumes. Without this, the agent has to reason from scratch every session about what's allowed.

## B4. What changes about `docs/authoring.md`

The doc is now a **procedure spec for the agent**, not a how-to for humans. Concrete shape:

```
docs/authoring.md  (v2)
├── 0. Audience: this document is consumed by the LLM author agent.
├── 1. Goals invariants (what every story must satisfy).
├── 2. The author-loop procedure (call agent_brief → propose spec → call author_loop.py → handle response).
├── 3. Failure modes and recovery (what the agent does on each lint failure category).
├── 4. Override discipline (B2.2).
├── 5. Escalation criteria (when to stop trying and ask the human).
├── 6. Forbidden actions (no manual stories/*.json edits, no skipping the gauntlet, no inventing words outside vocab_state.json).
└── 7. Prompt fragments (the system prompt the human invokes the agent with — kept in the same doc so it can be edited as a single artifact).
```

Every section is structured in a way that an LLM reading it can *act on*: numbered procedures, JSON examples, error-message → action tables.

## B6. Updated open questions (replaces §A6's three additions)

These supersede A6 questions 6–8:

6. **Reviewer model.** Different model family (e.g. Claude author + GPT reviewer) vs. same family with a different persona prompt? The former is more robust against shared blind spots; the latter is operationally simpler.
7. **Escalation cadence.** When the agent's 3-round retry fails, what's the SLA on you (the human) responding? This determines whether `author_loop.py` should support a "queue for human review" path or a "halt the entire pipeline" path.
8. **Spec field discipline.** Are `intent`, `scene_class`, `anchor_object`, and per-sentence `role` (B2.3) acceptable as required fields? They cost the agent a few hundred tokens per story but provide the upstream structure the audit found missing.
9. **Author-agent identity.** Is the author always *the same agent instance* across stories (with continuity via `agent_brief.py`'s `previous_3_stories_summary`), or is each story authored by a fresh instance with no episodic memory? The latter is more reproducible but makes motif-drift harder to fight.
10. **Is human authoring still allowed?** If yes, the dual-mode complicates `docs/authoring.md`. If no, that's a clean architectural commitment but worth being explicit about.

## B7. Honest reframing of the project

If the human author is fully out of the loop except for escalations and strategy, the project changes character:

- **The corpus becomes a function of the lints + reviewer + north-stars + tools**, not of any individual's literary voice.
- **Quality now depends on the rule set, not the writer.** The audit becomes the *most important artifact in the repo* — every defect it found needs a corresponding lint rule or reviewer prompt clause, because there's no human reading every story.
- **Story 1's voice becomes the entire corpus's voice ceiling.** The agent imitates the established corpus; without strong early stories there is no anchor. The §1.5 north-stars become the *only* things keeping voice from drifting.
- **Failure mode becomes silent drift, not loud bugs.** A defect category neither the lints nor the reviewer name will accumulate forever. The audit cycle (read all 56 stories with fresh eyes) becomes a recurring requirement, not a one-time event. **Schedule it: every N stories or every M weeks, the human reads everything and surfaces new defect categories.**

The project *can* run with a fully agentic author. But it requires accepting that "literary quality" is now an emergent property of explicit rules, not of someone caring. That's a real trade-off — worth being honest about before committing to it.
